## 官方博客

### Anthropic Engineering · Claude Code 质量问题复盘

Anthropic 公开了过去一个月影响 Claude Code 性能的三项变更根因，均已于 4 月 20 日修复。

**问题一：推理效力静默降级（3月4日）**

为解决 Opus 4.6 在 high 模式下偶发的超长延迟，团队将默认推理努力从 `high` 静默改为 `medium`。用户随即反馈 Claude Code 感觉"变笨了"。尽管团队陆续推出了 UI 提示和效力选择器，大多数用户仍保持了 medium 默认值。4 月 7 日恢复为 `high`，Opus 4.7 起更升级至 `xhigh`。

**问题二：缓存优化 Bug（3月26日）**

本意是：session 空闲超过 1 小时后，清除一次旧的 thinking 块以节省恢复费用。实际行为：触发阈值后，**每一轮**都清除 thinking 历史，导致 Claude 在后续对话中丧失对先前决策的记忆，表现为遗忘、重复、以及奇怪的工具调用选择。因两个内部实验的干扰，该 bug 在内部测试中难以复现，历时超过一周才定位。4 月 10 日修复。值得一提的是，在代码审查场景中，用完整上下文的 Opus 4.7 发现了这个 bug，而 Opus 4.6 没有。

**问题三：冗长抑制提示（4月16日）**

系统提示新增一行：`"工具调用间保持 ≤25 词。最终响应保持 ≤100 词，除非任务需要更多细节。"` 在更宽泛的评估集上，该改动导致代码质量下降约 3%，4 月 20 日撤回。

三项变更作用于不同的流量分片，叠加后呈现出"广泛、不一致降级"的表象，增加了排查难度。Anthropic 已重置所有用户使用限额，并承诺加强系统提示变更的 eval 覆盖与审查流程。

🔗 https://www.anthropic.com/engineering/april-23-postmortem

### Anthropic Engineering · Managed Agents：脑手分离架构解析

Anthropic 工程博客深度拆解了 Managed Agents 从"全耦合单容器"到"脑手分离"的架构演进。

**旧设计的问题**

早期将 Claude 的 harness（大脑）与沙箱执行环境（双手）放在同一个容器内。容器即"宠物"——一旦崩溃，整个 session 丢失；调试时需要进入容器内部，而容器又持有用户数据，安全边界模糊。此外，每个 session 都需要先完整拉起容器才能开始推理，首 token 延迟高。

**新设计：三组件解耦**

- **Session（事件日志）**：持久化存储在容器之外，任意 harness 崩溃后执行 `wake(sessionId)` 即可从断点恢复
- **Brain（harness）**：变为无状态服务，通过统一接口 `execute(name, input) → string` 调用任意工具，不再感知底层是容器、MCP 工具还是其他资源
- **Sandbox（执行环境）**：变为"牲口"而非"宠物"，随时可以销毁重建，credentials 通过 OAuth vault 代理注入，沙箱内运行的不可信代码永远接触不到凭据

**实测效果**：p50 首 token 延迟降低约 **60%**，p95 降低超 **90%**。

### Claude Blog · Managed Agents 新功能：自托管沙箱与 MCP 隧道

（2026年5月19日）

Managed Agents 开放两项新能力，主要面向有数据合规要求的企业：

**Self-hosted sandboxes（公开 beta）**

Orchestration 逻辑继续运行在 Anthropic 服务器，工具执行转移到客户自有基础设施或受支持的托管沙箱提供商（Cloudflare、Daytona、Modal、Vercel）。企业代码和文件不需要离开自己的网络边界，安全策略、审计日志均由客户侧控制。

**MCP tunnels（研究预览）**

通过一个出向连接将内网 MCP 服务器（私有 API、内部数据库、知识库等）代理到 Managed Agents，无需开放公网端口，全程流量加密。Messages API 也同步支持 MCP tunnels。

🔗 https://claude.com/blog/claude-managed-agents-updates

## 播客

### AI & I by Every · 《我们用 AI 自动化了一切，员工却增加了三倍》

（2026年5月27日）

> "The further away an agent gets from a human, the less valuable it is."

Every CEO Dan Shipper 与 COO Brandon 聊了一个反直觉的现象：他们的公司从 4 人增长到 30 人，还在继续招聘——尽管 AI 工具已经渗透到每一个工作流程里。

**核心论点：AI 让昨天的专家能力变廉价，反而创造了更多需求**

AI 用所有人的历史输出训练，让任何人都能用 prompt 写代码、出报告、做设计。结果是：非专家开始越权做原本属于专家的事，产出大量"差不多对但没完全对"的内容。这些内容需要专家来：

1. 审核并修到真正能用
2. 搭建系统让这些产出可靠地流转
3. 用这些工具做以前根本不可能做的事（Every 的工程师一个人用两个月搭完了整个 inbox 功能）

这就是"自动化越多，工作越多"的悖论——不是工作消失了，而是工作的形态变了，对专家判断的需求反而上升。

**关于裁员潮的判断**

Dan 对近期大量"借 AI 名义裁员"的公告持怀疑态度：那些高调宣布"裁了 20% 员工、业务反而更好"的公司，更可能的解读是公司本身遇到了经营问题，AI 只是个说辞。大公司在重组时确实会有岗位变化，但把这包装成"AI 进步"并不诚实。

**一句结语**

"If you just ride the models — when new models come out, learn to use them for the stuff that you do — you're going to be fine."
